# 03. UI 인터랙션 모델 — Progressive Form

> **스코프**: 워크벤치의 사용자 대면 인터랙션. 컴포넌트 트리, 상태 시각화 규칙, 핵심 시나리오.
> **기본 패러다임**: **Progressive Form** — "양식 → 채우기"의 역순. 초안이 먼저 등장하고, 비어있거나 추정된 곳만 사용자가 채운다.
> **원칙**:
> 1. 입력 마찰 최소화 — 한 턴 1개 질문, 추론 가능한 건 안 묻고 보여주기만
> 2. 모든 문장은 출처가 보인다 — 사용자가 신뢰하려면 어디서 왔는지 보여야 함
> 3. 사용자는 언제든 개입 가능 — 추정값·골격·근거 모두 인라인 수정 가능
> 4. UI는 런타임 에이전트의 상태를 1:1 반영 — `01-agents.md`의 `annotation.status`가 UI 색상의 근원

## 0. 전체 레이아웃 (3패널 하이브리드)

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  [헤더]  autodoxc · 외국인 고용 계획서 — 미저장        [저장] [내보내기]   │
├────────────────────┬─────────────────────────────────┬───────────────────────┤
│                    │                                 │                       │
│   [채팅 패널]      │      [문서 캔버스]              │   [근거·골격 패널]    │
│   (좌측, 360px)    │      (중앙, flex)               │   (우측, 380px 토글)  │
│                    │                                 │                       │
│   - 자유 입력      │      - Progressive Form         │   - 골격 트리         │
│   - 에이전트 응답  │      - 문단별 상태 색상         │   - 수집된 근거       │
│   - 인라인 질문 1개│      - 빈/추정 슬롯 클릭        │   - 사용자 라이브러리 │
│   - 첨부 업로드    │      - 인라인 편집              │   - 외부 인용         │
│                    │                                 │                       │
│                    │                                 │  [패널 닫기 ▶]        │
└────────────────────┴─────────────────────────────────┴───────────────────────┘
```

**기본 가시성**
- 좌·중 패널: 항상 노출
- 우 패널: 기본 닫힘, 사용자가 근거 클릭 또는 [근거 보기] 누를 때 열림 — **새 사용자에게 시각적 과부하 방지**

**반응형**
- `>= 1280px`: 3패널
- `1024~1279px`: 우 패널 모달/사이드시트로 강등
- `< 1024px`: 캔버스 전체화면, 채팅과 근거는 하단 시트 (탭 전환)

## 1. 시작 화면 (Entry State)

세션 시작 시 캔버스에는 아무것도 없고, 좌측 채팅에 단 한 줄의 프롬프트.

```
┌────────────────────────────────────────┐
│                                        │
│   안녕하세요. 어떤 문서를 쓰실까요?   │
│   사건이나 상황을 자유롭게 말씀해      │
│   주세요. 양식 파일이 있으면 던져      │
│   주셔도 됩니다.                       │
│                                        │
├────────────────────────────────────────┤
│  [예시: 외국인 고용 계획서를 써야  ]   │
│  [      해. 5축 가공 기술자 한 명 ]   │
│                                        │
│  [📎 파일 첨부] [↑ 전송]               │
└────────────────────────────────────────┘
```

**금지 사항**
- 시작 시점에 양식 선택 드롭다운, 카테고리 트리, "어떤 종류의 문서인가요" 라디오 버튼 등 강제 분류 X. GPTs식 폼의 잔재.
- 사용자가 한 줄로 적으면 즉시 #1a가 동작. 신뢰도 낮으면 그때 후보 채팅 메시지로 제시.

**드래그앤드롭 영역**
- 빈 캔버스 전체가 드롭존 — `.docx`, `.pdf`, `.hwp`, `.hwpx`, 이미지 받음
- 드롭 즉시 DA4 파서 + #1a + #1b 가동, 캔버스에 골격 윤곽이 깜빡이며 등장

## 2. 중앙 패널: 문서 캔버스 (Progressive Form 본체)

### 2.1 문단 상태 시각화 — 5가지

`DraftParagraph.annotations.status` (01-agents.md 정의)를 시각적으로 1:1 매핑:

| Status | 시각 표현 | 의미 | 사용자 액션 |
|---|---|---|---|
| `confirmed` | 기본 본문(검정·흰배경) | 사용자 사실에 근거, 변경 없음 | 일반 편집 |
| `inferred` | 옅은 노랑 배경, 점선 밑줄 | 시스템 추정값, 사용자 미확인 | 클릭 → assumption 보기/수정 |
| `defaulted` | 옅은 회색 본문, 작은 ⓘ 아이콘 | 스마트 디폴트(통계·관행) | 호버 → 출처, 클릭 → 수정 |
| `evidence_backed` | 기본 본문 + 우측 가장자리 파란 막대 | 외부 근거 인용 | 막대 클릭 → 우 패널에 출처 펼침 |
| `empty` | 점선 박스 + `[[필드명]]` 자리표시자 | 채워야 함 | 클릭 → 인라인 입력 |

> 시각 표현은 디자인 토큰 변경 가능. 핵심은 **5개 상태가 시각적으로 구분**되어야 한다는 규약.

### 2.2 컴포넌트 트리

```
<DocumentCanvas>
  <CanvasHeader>           {docType, lastSavedAt, draftReadiness}
  <ProgressBar>            {confirmed/inferred/empty 비율}
  <SectionList>
    <Section>              {sectionId, title, status: complete|partial|empty}
      <SectionHeader>      {title, [추가/삭제/이동 메뉴]}
      <ParagraphList>
        <Paragraph>        {annotations.status에 따라 5가지 스타일 중 하나}
          <Token text|EmptySlot|InferredSpan|EvidenceCitation />
      <AddParagraphHint>   {sectionId, 호버 시 "+ 문단 추가"}
    <AddSectionHint>       {위치, 호버 시 "+ 섹션 추가"}
</DocumentCanvas>
```

### 2.3 인라인 편집 — 핵심 인터랙션

**EmptySlot 클릭**
```
원래:  ...우리 회사 [[회사명]]는 5축 가공 분야에서...
       ↓ 클릭
편집:  ...우리 회사 [입력박스 활성, focus] 는 5축 가공...
       ↓ Enter
완료:  ...우리 회사 ABC공업(주)는 5축 가공 분야에서...   ← confirmed로 전환
       ↓ 동시에
       채팅 패널에 시스템 메시지: "회사명을 'ABC공업(주)'로 기록했어요."
```

**InferredSpan 클릭**
```
[옅은 노랑] "최근 1년간 6회의 채용 시도가 있었으며"
       ↓ 클릭 — 팝오버 등장
┌─────────────────────────────────┐
│  추정 근거                       │
│  · 사용자 입력에서 "여러 번"     │
│    표현 → 6회로 추정             │
│                                  │
│  [그대로 사용] [수정] [질문 받기]│
└─────────────────────────────────┘
```
- `[그대로 사용]`: status를 `confirmed`로 승격
- `[수정]`: 인라인 편집 모드
- `[질문 받기]`: 채팅 패널에 GapAnalyzer가 다시 #3 호출해 질문 1개 생성

**EvidenceCitation 클릭**
```
"...행정절차법 제22조 제3항 [1]에 따라..."
       ↓ [1] 클릭
       우 패널 자동 오픈 → 해당 evidence 카드로 스크롤
```

### 2.4 상단 진행률 바

```
┌────────────────────────────────────────────────┐
│ 확정 62%   ▓▓▓▓▓▓▓░░░  추정 28%   빈칸 10%   │
└────────────────────────────────────────────────┘
```
- 사용자가 "얼마나 채워졌는지" 한눈에. 빈칸이 0이면 저장/내보내기 활성화.
- 클릭 시 빈칸 또는 추정 섹션으로 점프 (다음 빈 슬롯 순환 이동)

### 2.5 골격 변경 UX

- 섹션 헤더의 `⋮` 메뉴: `이름 변경` / `위로/아래로 이동` / `삭제` / `섹션 추가`
- 변경은 즉시 `SessionState.skeleton` 업데이트 → #8 학습 후보 큐에 적재 (저장 시점에 확정 학습)
- 사용자가 섹션을 삭제했다 복구 가능 (스택 5단계)

## 3. 좌측 패널: 채팅 (대화 진행자)

### 3.1 역할
- 사용자 자유 입력 누적 (FactsExtractor 입력 원천)
- 시스템 결정 알림 (assumption 생성, 골격 채택, 근거 수집 결과 등)
- **인라인 1개 질문 표출** — GapAnalyzer의 `next_question`

### 3.2 메시지 종류

| 종류 | 발신 | 형태 | 캔버스와의 관계 |
|---|---|---|---|
| User free text | 사용자 | 일반 채팅 버블 | FactsExtractor가 처리 → 캔버스에 반영 |
| System notice | 시스템 | 회색 옅은 박스, 짧음 | "회사명 'ABC' 기록", "유사 양식 매칭" 등 |
| Inline question | GapAnalyzer | 강조 박스 + 짧은 답변 인풋 | 답하면 캔버스의 해당 슬롯 채워짐 |
| Assumption announcement | 시스템 | 노란 가장자리 박스 + "수정" 링크 | 캔버스의 inferred 문단으로 스크롤 |
| Evidence summary | EvidenceRetriever | 파란 가장자리 박스 + 출처 K개 | 우 패널로 점프 |
| Reviewer warning | SelfReviewer | 빨간 가장자리 박스 | 캔버스의 issue 위치로 점프 |

### 3.3 Inline Question UX — 핵심

```
[시스템 박스]
┌────────────────────────────────────────────┐
│  한 가지만 확인할게요.                      │
│                                            │
│  ❓ 채용 시도 비용은 얼마였나요?            │
│      (개략값이면 충분합니다)               │
│                                            │
│  [입력 박스 ____________________] [건너뜀]  │
│                                            │
│  ⓘ 왜 묻나요? "고용 사유" 섹션에서         │
│    국내 채용 노력의 정량 근거가 필요합니다.│
└────────────────────────────────────────────┘
```

**규약**
- 한 턴에 **단 하나**의 질문만 (`GapAnalyzer.next_question` 직역)
- 항상 `[건너뜀]` 가능 → status는 `empty` 유지
- `ⓘ 왜 묻나요?` 토글 — 사용자 신뢰 확보 + 무엇 때문에 필요한지 투명성
- 답하면 즉시 캔버스 해당 문단의 status 전환 + 다음 질문 생성 (있을 때만)

### 3.4 입력 UX
- 멀티라인 자동 확장
- 파일 첨부는 언제든 가능 (드래그 또는 📎)
- `/` 입력 시 단축 명령 메뉴: `/규모 변경`, `/관할기관`, `/저장`, `/내보내기`

## 4. 우측 패널: 근거·골격 (토글)

기본 닫힘. 사용자가 근거 막대 클릭, [근거 보기] 버튼, 골격 트리 보기 등으로 열림.

### 4.1 탭 구조

```
┌──────────────────────────────┐
│ [근거] [골격] [라이브러리]  │
├──────────────────────────────┤
│                              │
│   탭 본문                    │
│                              │
└──────────────────────────────┘
```

### 4.2 근거 탭 (Evidence)
- 수집된 `Evidence[]` 카드 리스트, 타입별 아이콘
- 카드 클릭 시 캔버스의 `applied_to` 문단으로 스크롤 + 하이라이트
- `[근거 다시 찾기]` 버튼 — 해당 logic 노드에 대한 retrieval 재실행
- 사용자가 외부 출처 직접 추가 가능 (URL 또는 메모 첨부)

### 4.3 골격 탭 (Skeleton)
- `SkeletonNode` 트리. 각 노드에 진행 상태 뱃지(complete/partial/empty)
- 노드 클릭 → 캔버스 점프
- 노드 드래그로 재정렬
- 노드 우클릭: 삭제·복제·라이브러리 저장
- 상단에 `[골격 출처]`: composition_meta 요약 (예: "공식양식 70% + 라이브러리 20% + LLM 10%")

### 4.4 라이브러리 탭 (User Library)
- DA2 개인 라이브러리에서 사용자가 보유한 골격 목록
- 같은 `doc_type_id`의 변형들 비교 (diff view)
- 특정 골격을 현재 세션의 골격으로 강제 채택 가능 — `[이 골격으로 작성]`
- 공용 승격 가능 여부 표시 (`stats.usage_count` 진행률)

## 5. 핵심 인터랙션 시나리오

### S1. 처음 사용자 — 한 줄 입력 → 첫 초안

```
[채팅] 사용자: "외국인 고용 계획서 써야 해. 5축 가공 기술자 한 명."
        ↓
        [좌] "외국인 고용 계획서로 인식했어요. 골격을 구성 중입니다..." (1초)
        [중] 캔버스에 5개 섹션 윤곽 스켈레톤 등장 (회색 라인)
        ↓ (3~5초)
        [중] 섹션별로 초안 점진 등장 — 빈 슬롯·추정값·근거 인용 마킹 포함
        [상단] 진행률: 확정 18% · 추정 32% · 빈칸 50%
        [좌] "한 가지만 확인할게요. 채용 시도 횟수는 몇 회인가요?"
```

핵심: 추가 질문 없이 **한 줄로 초안 50% 완성**. 사용자가 GPTs식 4개 질문을 답하는 노동 0.

### S2. 첨부 양식 드래그

```
사용자가 캔버스에 .hwp 양식 파일 드롭
        ↓
        [중] "양식 분석 중..." 로딩 (3~10초, 포맷에 따라)
        ↓
        [중] 첨부 양식 그대로의 골격으로 캔버스 재구성
        [좌] "첨부하신 양식을 골격으로 채택했습니다. 양식의 [신청인 정보] 부분부터 채워주세요."
        [우] 골격 탭이 열려 출처: "사용자 첨부 (foo.hwp)" 100%
```

핵심: 첫 메시지에서 사용자가 보여준 4Q 분석본을 던지면 그게 그대로 골격이 됨. 미리 분석 안 해도 됨.

### S3. 인라인 질문 → 답변 → 캔버스 갱신

```
[좌] "채용 시도 비용은 얼마였나요?" [입력박스] [건너뜀]
사용자: "약 1500만원"
        ↓
        [좌] 시스템 박스: "비용 1,500만원으로 기록"
        [중] "고용 사유" 섹션의 비용 자리표시자 → "약 1,500만원" + 옅은 노랑 잠시 → confirmed
        [좌] 다음 질문 (또는 모든 질문 소진 시 "이제 검토만 남았어요")
```

### S4. 추정값 수정

```
[중] 옅은 노랑 문단 "최근 1년간 6회의 채용 시도가 있었으며"
사용자 클릭
        ↓
        팝오버: 추정 근거 + [그대로 사용] [수정] [질문 받기]
사용자: "수정" → 인라인 편집 → "최근 8개월간 4회"
        ↓
        문단 status confirmed로 전환, 노란색 해제
        [좌] 시스템 박스: "채용 시도: 8개월 4회로 정정"
        ↓
        다른 문단 중 이 값에 의존하던 곳도 자동 재생성 (예: "단기간 집중 노력"이라는 표현 등)
```

### S5. 저장/확정 → 학습기 트리거

```
사용자 [저장] 클릭
        ↓
        SelfReviewer #7 한 번 더 → 빈칸 K개·warning M개 표시
        ↓ 사용자 확인
        SkeletonLearner #8 비동기 트리거 (개인 라이브러리 누적)
        ↓
        세션 상태 "확정"
        [내보내기]: docx / pdf / hwp 선택
```

## 6. UI 상태 머신

```
[Idle / Entry]
     │ (사용자 입력 또는 첨부)
     ▼
[Identifying]            ← #1a 동작 중, 좌측에 짧은 알림
     │ (성공)
     ▼
[Composing skeleton]     ← #1b, 캔버스에 윤곽 스켈레톤 페이드인
     │
     ▼
[Drafting initial]       ← #2,3,4,5,6 직렬/병렬 실행, 캔버스에 점진 채움
     │ (초안 완료)
     ▼
[Editing]  ◄────────┐    ← 메인 상태. 사용자 자유 편집 + 인라인 Q&A
     │              │
     │ (사용자 변경)│
     │              │
     ▼              │
[Re-drafting partial]┘   ← 영향받은 문단만 재생성 (전체 X)
     │
     │ (사용자 저장)
     ▼
[Reviewing]              ← #7 SelfReviewer 종합 점검
     │
     ▼
[Saved]                  ← #8 비동기 학습 시작
     │
     │ (사용자 재편집 또는 내보내기)
     │
     └──→ Editing 또는 종료
```

**상태별 UI 표시**
- Identifying / Composing: 캔버스 중앙에 스피너 + "골격 구성 중"
- Drafting initial: 캔버스 섹션이 위에서 아래로 페이드인
- Editing: 헤더에 "변경 사항 있음" + 저장 버튼 강조
- Re-drafting partial: 영향 문단에 미세한 깜빡임 (눈에 거슬리지 않게)
- Reviewing: 사이드시트에 issue 리스트, 사용자가 일괄 확인
- Saved: 헤더 "마지막 저장 N분 전"

## 7. 접근성·키보드·반응형

### 키보드 단축키
- `Ctrl/Cmd + S`: 저장
- `Ctrl/Cmd + K`: 빈 슬롯 점프 (다음/이전은 `Tab` / `Shift+Tab`)
- `Ctrl/Cmd + /`: 채팅 명령 메뉴
- `Esc`: 인라인 편집 취소
- `Ctrl/Cmd + Enter`: 인라인 질문 답변 전송

### 접근성
- 5가지 상태 색상은 색약 대비 검증 + 항상 텍스트/아이콘 보조 (색에만 의존 X)
- 모든 인라인 상태 변화는 `aria-live` 영역에서 음성 알림 (예: "문단이 확정 상태로 변경")
- 빈 슬롯은 `role="textbox"` + `aria-label` 명시

### 반응형 핵심 룰
- 모바일에서 채팅·캔버스·근거가 **동시 노출되지 않음** — 하단 탭으로 전환
- 캔버스 폭은 모바일에서도 가독성 유지 (행간·여백 조정)
- 첨부 파일 드롭은 모바일에서는 명시적 버튼만 (드래그 X)

## 8. 무엇을 하지 않는가 (반패턴)

UI 설계에서 명시적으로 피하는 것:

| 반패턴 | 이유 |
|---|---|
| 시작 시 양식 카테고리 트리·분류 드롭다운 | GPTs식 마찰. 자연어 입력만으로 시작 가능해야 함 |
| 한 번에 여러 빈칸을 채우는 폼 모달 | "역양식"의 정신과 어긋남. 빈 슬롯은 캔버스 안에서 채워야 함 |
| 추정값에 색상 없이 본문에 섞기 | 사용자가 무엇이 시스템 추정인지 모르면 신뢰 잃음 |
| "AI가 답을 못 찾았어요" 같은 막연한 빈 상태 | 항상 다음 행동(질문/추정/건너뜀)을 명확히 제시 |
| 채팅 패널에 긴 단락 응답 | 채팅은 결정·질문·알림용. 본문은 캔버스에 |
| 근거 패널을 기본 노출 | 신규 사용자에 인지 과부하. 필요 시 열림 |
