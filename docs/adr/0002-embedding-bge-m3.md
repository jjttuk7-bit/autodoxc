# ADR 0002: 임베딩 모델 — BGE-M3 자체 호스팅 (콜드스타트는 OpenAI API)

**상태**: Accepted
**날짜**: 2026-06-12
**결정자**: lead-architect, with user/PM

## 맥락

DA3 RAG 인덱스 + 사용자 골격 라이브러리 검색의 임베딩 계층. 한국어 행정문서가 주 코퍼스이며 PII가 포함될 수 있어 외부 API 호출 노출을 최소화해야 함. 동시에 콜드스타트 단계의 GPU 호스팅 부담을 분리해야 함.

## 결정

**2단계 전략**:

### Phase B0~B1 (콜드스타트)
- **OpenAI `text-embedding-3-large`** API 사용
- 이유: GPU 인프라 없이 즉시 시작 가능, 인증·계약은 D9 호스팅과 별도
- 차원: 3072 — pgvector 인덱스 크기 부담 인지하되 1차 코퍼스 수백~수천 건 규모에서는 허용
- PII는 EXT 게이트(`02-data-assets`)를 통해 사전 추상화 후 임베딩 호출 — 원본 PII가 외부에 나가지 않게

### Phase B2 이후 (전환)
- **BGE-M3 자체 호스팅**으로 마이그레이션
- 차원: 1024 — 인덱스 크기 ⅓로 축소
- GPU 인스턴스: AWS Seoul `g5.xlarge` 또는 동등 (D9와 연동)
- 마이그레이션 절차: 전체 재인덱싱 잡, 신·구 인덱스 듀얼 라이브 후 컷오버

**근거**:
- BGE-M3는 한국어 retrieval에서 OpenAI 동등 또는 우위, 차원이 1/3 → 운영 비용·지연 우수
- 자체 호스팅으로 PII 외부 노출 0
- 오픈소스(Apache 2.0)라 도메인 fine-tune 자유 — 누적 데이터로 행정문서 특화 가능

## 대안

- **Solar embedding-1-large (국내)**: 한국어 강점이나 API 종속·차원 큼(4096). 마이그레이션 자유도 낮음
- **OpenAI 단일 (전환 없음)**: 차원·비용·PII 노출 부담이 누적
- **BGE-M3 첫날부터**: GPU 인프라 운영 부담이 콜드스타트 작업을 지연

## 결과

**영향 모듈**:
- `src/data/rag/embedder.ts` — 임베딩 인터페이스 (공급자 중립)
- `src/data/rag/indexer.ts` — 차원 설정 동적
- `src/data/rag/migration/` — 마이그레이션 잡 (Phase B2 작성)

**마이그레이션 트리거 조건**:
- (1) 운영 비용에서 OpenAI 임베딩 호출 월 $200 초과 또는
- (2) PII 외부 노출 우려 정책 강화 또는
- (3) Phase B2 진입 시 — 중 가장 빠른 시점

**듀얼 라이브 절차**:
1. 신규 인덱스를 BGE-M3로 생성, 백필 잡 가동
2. 검색 코드는 신·구 인덱스 둘 다 조회, 결과 비교 로깅
3. 1주 비교 후 신규로 컷오버, 구 인덱스 폐기

**회귀 검증**: M6가 retrieval 정확도 평가 fixture로 baseline 측정. 마이그레이션 후 동일 fixture로 회귀.

**연동 결정**: D9에서 BGE-M3 운영 시 GPU 인스턴스 비용을 인프라 IaC에 반영.
